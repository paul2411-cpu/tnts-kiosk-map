========================================================================
1. OVERALL ASSESSMENT
========================================================================

The TNTS software project itself is stronger than the manuscript that
describes it.

After examining the manuscript text, the embedded images, and the current
project code, the main conclusion is:

- The manuscript contains a real TNTS core.
- Many embedded screenshots do match the TNTS system.
- The manuscript text is not yet clean, consistent, or academically ready.
- Some sections overstate features.
- Some sections underdescribe the system's strongest technical work.
- Several parts appear to be copied or left over from another project.

In short:

The project looks real and technically substantial, but the manuscript
currently weakens the project because the write-up is inconsistent,
partially contaminated by unrelated content, and unclear in scope and stage.

4.8 Feature overstatement or inaccurate framing
------------------------------------------------------------------------

Some descriptions in the manuscript do not align perfectly with the current
system.

Important examples:

- The manuscript uses "real-time mapping" language.
  The current system loads the published live map state and polls for a newer
  published version. It is not real-time indoor positioning or live user

  The current public events page is a schedule-filtered card interface, not a

  especially student-leader style users with limited access to events and
  announcements, not a separately engineered end-user admin application.

Why this is a problem:

- It risks overclaiming.
- A reviewer comparing the document to the actual system may notice the
  mismatch.

========================================================================

The project should be framed not merely as:

"a kiosk with a 3D map, facilities, events, announcements, and feedback"

"a model-aware campus navigation and information platform with a published
3D map runtime, searchable building and room catalog, route and text-guide
support, event and facility map integration, and an admin-side workflow for
importing, editing, versioning, and publishing campus map data."

That framing better matches the real system.

11. STRONGER SYSTEM DESCRIPTION THAT THE MANUSCRIPT COULD USE
========================================================================

Below is a cleaner technical description of TNTS based on the actual system:

The Tanza National Trade School Navigator is a kiosk-oriented campus
navigation and information platform built with PHP, MySQL, JavaScript,
Three.js, and Blender-generated GLB assets. On the public side, it provides
an interactive campus map with 2D and 3D viewing modes, destination search,
building and room detail panels, route guidance, facilities browsing, event
integration, announcement display, and feedback submission. On the
administrative side, it supports role-based access to building, room,
facility, event, announcement, feedback, and account management modules,
along with a model-aware map import and publishing pipeline. Administrators
can scan and synchronize GLB map models, maintain destination metadata,
author routes and guide text, place event-specific areas on the map, export
updated model states, and publish a selected model as the live campus map
used by the public kiosk runtime.

This description is much closer to what the actual codebase supports.
